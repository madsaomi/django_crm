# 🤖 EduCRM AI Architecture Guide (Внутренняя документация для ИИ)

Привет, будущий ИИ-ассистент или разработчик! Этот документ создан для того, чтобы ты мог моментально понять бизнес-логику, структуру базы данных и архитектурные решения этой образовательной CRM-системы.

Проект написан на **Django 6.0** и использует интегрированную базу данных **SQLite** (файл db.sqlite3). Дизайн опирается на кастомный Bootstrap 5 с Glassmorphism, а PWA-интеграция делает его похожим на мобильное приложение.

---

## 🏗️ Общая Архитектура
Паттерн: Стандартный Django MVC (Fat Models, стандартные Views на основе CBV - Class Based Views).
Особенность: По умолчанию база данных и авторизация завязаны на стандартном `User` Django, но сущности `Teacher` и `Manager` связаны с `User` через связь OneToOne.

---

## 📦 Описание Приложений (Apps)

### 1. `students` (Ученики)
**Суть:** Хранит базовую информацию о клиентах (учениках) и привязывает их к менеджерам (кто привел/ведет клиента).
- **Модель `Student`**: `name`, `phone`, `manager` (FK), `is_active`.
- **Логика**: Ученики не имеют прямого доступа к системе (пока что), это просто записи для учёта. Если ученик "удаляется", лучше менять `is_active = False` (Soft Delete).

### 2. `teachers` (Преподаватели)
**Суть:** Преподаватели, ведущие уроки в группах.
- **Модель `Teacher`**: `user` (OneToOne к `User`), `name`, `phone`, `specialization`.
- **Логика**: При создании преподавателя создаётся `auth.User`, чтобы препод мог логиниться и отмечать посещаемость. Текущие преподы: Диёра Рашидова (English), Камолиддин Хошимов (English), Шахзод Абдуллаев (IELTS), Нодира Усманова (Math/SAT).

### 3. `managers` (Менеджеры)
**Суть:** Отдел продаж или администраторы, которые регистрируют студентов и платежи.
- **Модель `Manager`** (или Staff): связь с `auth.User`. Менеджеры назначаются студентам.

### 4. `groups` (Группы и Расписание - ❤️ Ядро Системы)
**Суть:** Управляет учебными группами, расписанием и привязкой студентов (Enrolment).
- **Модели**:
  - `Group`: Описывает группу (`course_type`, `teacher`, `schedule_type`: Четный/Нечетный день, `monthly_fee`, `max_students`, `room`, `color`). Свойство `is_full` проверяет вместимость группы.
  - `Enrollment`: Many-to-Many таблица связи ученика и группы с датой зачисления и статусом активности.
  - `SundayEvent`: Бронирование кабинетов по воскресеньям (разовые ивенты вне регулярного расписания `Group`).
- **💡 Особенности для ИИ (UI/JS)**: Страница расписания (`groups/schedule.html`) использует библиотеку Sortable.js для Drag & Drop. Перенос карточки триггерит AJAX запрос к `UpdateGroupScheduleView`, который меняет `schedule_type` и `time_slot`.

### 5. `attendance` (Посещаемость)
**Суть:** Журнал отметок.
- **Модель `Attendance`**: `student`, `group`, `date`, `status` (PRESENT/ABSENT).
- **💡 Особенности для ИИ (Оптимизация)**: В `apps/groups/views.py` в представлении `GroupDetailView` стоит оптимизированный алгоритм сборки журнала (через `date__in=dates` и выгрузку `list()`). **Критически важно** не допускать там N+1 запросов к базе, если будешь добавлять новые вычисления!

### 6. `payments` (Платежи)
**Суть:** Финансовый лог. 
- **Модель `Payment`**: `student`, `amount`, `date`.
- **Логика**: Здесь нет сложных транзакций или счетов (Invoices), только сырые логи оплат. Для расчета долга студента (Debt) необходимо будет вычислять: `(Ожидаемый доход от активных групп студента) - sum(Payment.amount)`.

### 7. `kpi` (Метрики Дашборда)
**Суть:** Аналитика и Дашборд. 
- Не содержит тяжелых моделей. Основная логика лежит в `kpi/views.py` (или `kpi/services.py`), где суммируются ученики и доходы (`aggregate(Sum('amount'))`).

## 🎨 Визуальные Решения (CSS & UX)
- **Тёмная тема (Dark Mode):** Логика переключения написана в `base.html` (localStorage) и обрабатывается через CSS-селектор `[data-bs-theme="dark"]` в `style.css`.
- **Page Transitions:** Спрятан `Loader` (`#pageLoader`), который перекрывает экран при переходах между внутренними страницами (с задержкой 200ms) для создания эффекта SPA без перезагрузок. Не сломай его, если будешь добавлять атрибуты к ссылкам `<a>` (в JS игнорируются ссылки с `target=_blank` и хэшами).
- **PWA:** Проект зарегистрирован как PWA. Статика отдается через `templates/sw.js` (перехватчик) и `manifest.json`.

## ⚡ Что стоит внедрить в первую очередь, если пользователь попросит "Архитектурных улучшений"
1. **Переход на HTMX:** Замени AJAX-запросы и `redirect` при сабмите форм на элегантный HTMX-флоу, чтобы открывать формы создания платежа/ученика в Bootstap Offcanvas.
2. **Celery Tasks:** Добавь `tasks.py` для асинхронной рассылки сообщений о пропуске уроков.
3. **Repository/Service Layer:** Вынеси расчеты из Views в отдельные сервисные классы (например, `class StudentFinancialService: def get_debt(student) -> int`).
